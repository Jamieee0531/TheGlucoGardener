# PRD: Vision Agent - Multimodal Image Processing Module

> 版本: v0.1 | 日期: 2026-03-05 | 作者: Jamie (康金宇)

---

## 一、产品概述

### 一句话描述

Vision Agent 是慢病管理 chatbot 的图片处理模块，面向新加坡糖尿病患者，将用户上传的饮食照片、药物照片和化验报告自动转化为结构化健康数据，供下游 Agent 生成个性化建议。

### 背景与动机

- **所属项目**: SG INNOVATION Challenge - 面向新加坡糖尿病患者的 AI 远程健康管理平台
- **平台核心**: 游戏化健康管理系统，包含个人/群组聊天 chatbot、预警 Agent、任务发布 Agent、游戏交互等模块
- **Vision Agent 的角色**: chatbot 的"眼睛"。当用户在聊天中发送图片时，chatbot 的 triage 层识别到图片输入，路由到 Vision Agent 进行处理，返回结构化 JSON，再由 chatbot 的专家/陪伴等 Agent 基于这些数据给用户回复
- **为什么需要它**: 糖尿病患者的日常管理高度依赖饮食控制、用药规范和指标追踪。纯文字聊天无法处理用户拍摄的食物照片、药盒照片或纸质化验单，需要一个专门的多模态处理模块将图片信息"翻译"为 chatbot 能理解的结构化数据

### 目标用户

- **主要用户**: 新加坡糖尿病患者（核心为 2 型糖尿病）
- **使用场景**: 在 chatbot 聊天过程中上传图片
  - 三餐拍照打卡 → 想知道吃了什么、营养如何
  - 拍药盒/处方单 → 想确认药名、剂量、用法
  - 拍纸质化验单 → 想记录 HbA1c、血糖等关键指标
- **间接用户**: chatbot 的其他 Agent（专家 Agent、任务发布 Agent 等），它们消费 Vision Agent 输出的结构化数据

## 二、核心功能（MVP 范围）

### 必须有（Must Have）

1. **饮食照片识别**: 识别食物名称（侧重新加坡本地食物如 chicken rice、laksa、nasi lemak 等）+ 估算营养数据（热量、碳水、蛋白质、脂肪等）
2. **药物/处方照片识别**: 提取药名、剂量、频次；支持保健品多成分提取（如 ingredients 列表）
3. **化验报告识别**: 提取关键指标（HbA1c、空腹血糖、餐后血糖等），包括数值和单位
4. **非目标图片拒识**: 非饮食/药物/报告类图片（如自拍、风景照）返回明确的拒识结果，不胡乱解读
5. **多图片输入**: 支持用户一次发送多张图片，逐张处理并汇总结果
6. **统一结构化输出**: 所有场景输出标准化 JSON，包含场景类型、置信度、提取数据，方便下游 Agent 直接消费

### 最好有（Nice to Have）

1. **FoodAI API 集成**: 接入专业食物识别 API 提升新加坡本地食物的识别准确度（目前 API 审批 pending）
2. **置信度量化评估体系**: 建立标注数据集，对识别准确率进行量化测试
3. **更多本地化优化**: 扩展新加坡本地食物数据库、本地药物品牌覆盖

### 明确不做（Out of Scope）

- **不生成文字建议**: 不对识别结果进行解读或给出健康建议（这是 chatbot 专家 Agent 的职责）
- **不做数据存储**: 不负责将结果写入数据库或用户档案（这是 chatbot 层面的职责）
- **不处理语音输入**: 语音相关处理由 chatbot 的 MERaLiON SER 模块负责
- **不做下游分析**: 不基于图片内容生成用户回复，只输出结构化数据

## 三、用户流程

### 核心用户旅程

```
用户在 chatbot 中发送消息（可能包含图片 + 文字）
  → chatbot triage 识别到图片输入
  → 图片部分路由到 Vision Agent
  → Vision Agent 识别场景类型（饮食/药物/报告/未知）
  → 按场景提取关键信息
  → 返回结构化 JSON 给 chatbot
  → chatbot 将 Vision Agent 的 JSON + 用户文字 + 对话上下文等信息一并送给大模型，生成回复
```

### Vision Agent 内部流程

```
图片输入 → 校验 + base64 编码 → 场景分类 → 场景专属提取 → 数据校验 + 置信度评估 → 结构化 JSON 输出
```

> 注: 详细的内部节点设计、条件路由逻辑等工程细节见后续工程实现计划。

## 四、产品设计要点

### 核心机制

- **无状态处理**: Vision Agent 是纯函数式的——图片进、JSON 出，不维护对话状态或用户历史。这使它可以被任何上游模块调用，不产生耦合
- **场景分类 + 专属提取**: 先判断图片属于哪个场景，再用针对性的提取策略处理，而非用一个通用 prompt 应对所有情况
- **新加坡本地化**: prompt 层面针对新加坡本地食物、本地药物品牌做专门优化，这是区别于通用 VLM 直接调用的核心价值

### 设计原则

- **模型可替换**: VLM 层抽象为接口，当前用 Gemini 2.5 Flash，可随时切换到 SEA-LION 或其他模型，不影响上层逻辑
- **只做 Vision**: 职责边界清晰，不越界做 chatbot 该做的事
- **宁可拒识，不可胡说**: 对于无法识别或置信度低的图片，返回明确的"无法识别"而非编造结果，这在医疗场景下尤为重要

## 五、技术方向

| 层面 | 选型 | 理由 |
|------|------|------|
| 编排框架 | LangGraph | 状态图 + 节点 + 条件边，适合多分支处理流程 |
| VLM（视觉） | Gemini 2.5 Flash | 多模态能力强，支持多图片原生输入 |
| VLM（文本） | SEA-LION 27B | 主办方推荐模型，OpenAI-compatible API |
| 数据校验 | Pydantic v2 | 结构化输出定义 + 运行时校验 |
| 语言 | Python 3.11+ | 团队技术栈 |
| HTTP 客户端 | httpx | 异步支持，API 调用 |

> 注: 技术选型的细节和替代方案评估见后续工程实现计划。

## 六、成功指标

### MVP 阶段目标

- [ ] **比赛 Demo 可用**: 能在评委面前跑通三个场景（饮食、药物、报告）的完整流程
- [ ] **识别质量可验证**: 对测试图片集（目标 20 张）能正确提取核心信息（食物名称、药名、指标数值），无明显错误
- [ ] **置信度机制有效**: 低于阈值的识别结果能正确触发拒识/降级，不输出不可靠数据
- [ ] **集成 chatbot 稳定**: 接入 Bailey 的 chatbot 后能稳定运行，接口清晰，不出现阻塞或报错
- [ ] **测试覆盖充分**: 单元测试覆盖率 80%+，核心路径有集成测试

## 七、风险与待定项

| 风险/问题 | 影响 | 应对方案 |
|-----------|------|---------|
| FoodAI API 审批 pending | 新加坡本地食物识别依赖通用 VLM，准确度可能不够 | 先用 Gemini + SG-localized prompt 兜底，API 下来后作为增强 |
| SEA-LION API 限流（试用版 6 次/分钟） | 实时 demo 时可能触发限流 | RetryVLM 指数退避 + 降级到 Gemini |
| 新加坡本地食物多样性 | 通用 VLM 对本地食物识别不够精准 | 方案待定（候选：prompt 内嵌本地食物知识 / 本地 RAG / FoodAI API），需进一步评估 |
| 化验报告格式多样 | 不同医院/诊所的报告格式差异大 | MVP 阶段聚焦常见格式，标注 limitations |
| 与 chatbot 集成的接口定义 | 双方接口不匹配会阻塞集成 | 尽早与 Bailey 对齐 JSON schema 和调用方式 |
| 比赛截止时间紧（3/19） | 功能范围需严格控制 | 坚守 MVP 范围，Nice to Have 全部推后 |
